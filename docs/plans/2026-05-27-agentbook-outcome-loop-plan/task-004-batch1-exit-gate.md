# Task 004: Batch 1 exit gate (full 17-task gemma4_e4b sweep)

**depends-on**: task-003

## Description

Operator-run measurement task that closes Batch 1. Run the full 17-task hard sympy subset against gemma4_e4b × 4 runtime arms × k=3 with the lenient parser in place; archive results to `runs_v2.batch1/`; write `runs_v2.batch1/SUMMARY.md` with the 5 exit-gate measurements and the per-iid optimization signals. Decide whether to advance to Batch 2 (gates 1-4 pass and gate-5 threshold from task-001 holds) or to stop and root-cause (gate 5 negative).

This is an ops/measurement task — no BDD scenario. Its output is the entry-gate input for task-005 (Batch 2 has nothing to refine if `select_stuck(min_failure_count=3)` returns empty).

## Steps

### Step 1: Run the full sweep
- `cd experiments/agentbook-ab && uv run python -m pipeline.orchestrator --arms good good_synth good_loop good_multi_loop -k 3` against the 17 hard sympy tasks.
- Default `--workers` is fine; no rate-limit concern (Ollama is local).
- On completion, `mv runs_v2 runs_v2.batch1`.

### Step 2: Refresh outcomes log
- `cd experiments/agentbook-ab && uv run python -m pipeline.router` — regenerates `_oracle/outcomes_log.json` from the batch-1 archive (point its `runs_v2.batch1` path as needed per the existing bootstrap convention).

### Step 3: Compute the 5 gate measurements
- Read all measurements from `_oracle/outcomes_log.json` (never grep `runs_v2/` ad-hoc per best-practices.md §"Cross-batch invariants").
- **Gate 1** — All unit tests: `uv run python -m pytest experiments/agentbook-ab/ -q`. Expect 15 + 11 = 26 in `test_search_replace.py` plus the rest of the suite green.
- **Gate 2** — Truncated-edit notes drop: count `episode.notes` entries beginning with `no-block` containing `` ```edit `` in the Batch 1 archive vs the pre-flight archive. Expect `truncated_no_block_notes_post ≤ 0.2 × pre`.
- **Gate 3** — Doom-loop episodes drop: count episodes with ≥ 4 consecutive identical-prefix unparseable-edit turns in `runs_v2.batch1/`. Expect 0.
- **Gate 4** — No regression: every task resolved in the pre-batch-1 baseline (from `runs_v2.preflight/` OR the prior archive that produced K_pre = 15) is still resolved.
- **Gate 5** — Union holds or grows: `K_post ≥ K_pre` (Branches B/C) or `K_post == K_pre` (Branch A). Compute K from arm × sample union per the canonical `score.py` / `pipeline.router.evaluate_offline` definition.

### Step 4: Compute optimization signals
- Per-arm pass@3 delta vs the pre-batch baseline. Note any arm whose lift is concentrated (e.g., `good_loop` lifts but `good_multi_loop` flat → informs Batch 3 routing weights).
- Distribution of `diagnose_edit_block(text)` outcomes across the run. Tally counts of "missing closing triple-backtick", "missing ======= separator", "missing >>>>>>> REPLACE marker", "missing opening ```edit fence", "malformed edit block (catch-all)".
- Per-iid `notes`-count delta. List tasks where notes drop sharply but `resolved` stays False — these are Batch 2 cue-refinement candidates.

### Step 5: Write `runs_v2.batch1/SUMMARY.md`
- Table with the 5 gate rows: `#`, `Check`, `Threshold`, `Observed`, `PASS/FAIL`.
- Table with optimization signals.
- Final paragraph: explicit decision — "Advance to Batch 2" or "Stop and root-cause `<reason>`". Cite which gate failed if any.
- If Gate 5 negative: do NOT advance. Inspect iids that regressed against the recovered-tuple log (the test "test-file refusal still fires for a recovered (unfenced) block" pins the obvious failure mode; any other regression is the lenient parser silently applying a wrong edit).

### Step 6: Commit Batch 1
- Per cross-batch invariant: each batch is its own commit. Commit covers the task-002, task-003, and task-004 changes plus the archive metadata.

- `runs_v2.batch1/SUMMARY.md` exists and records all 5 gate rows with explicit `PASS`/`FAIL` resolutions against the thresholds.

- The lenient parser leaves no silent regression: `regression_count == 0` per Gate 4.
- Batch 1 lands as its own commit (per cross-batch invariant).
- If `select_stuck(model_slug="gemma4_e4b", min_failure_count=3)` against the new outcomes log returns empty AND gates 1-4 pass, the summary explicitly notes "Batch 2 deferred — nothing to refine" and the plan exits cleanly without proceeding to task-005.
